# Redis 캐싱으로 배운 Cold Start의 비밀

**작성**: 2026-04-23  
**주제**: Phase 4 성능 최적화 - Cold Start vs Warm Cache 분석  
**독자**: 백엔드 개발자, 성능 최적화에 관심있는 분들

---

## 🚀 들어가며: 숨겨진 성능 차이

N-QL Intelligence 프로젝트의 Phase 4에서 Redis 캐싱을 도입했습니다. 계획했던 목표는 **30.94ms → 20ms 달성**이었어요.

하지만 실제 결과는 예상을 뛰어넘었습니다.

```
1회차 (Cold Start):    53.81ms  ❌ 기대 이하
2회차 (Warm Cache):    36.05ms  ✅ 33% 개선!

GROUP BY sentiment:    54.85ms → 27.52ms  (49.8% 개선!)
```

**뭔가 이상합니다.** 

왜 1회차는 그렇게 느리고, 2회차는 갑자기 빨라졌을까요? 그리고 예상과 달리 GROUP BY가 가장 큰 개선을 보였을까요?

이 이야기는 **캐싱의 진정한 가치**를 깨달은 여정입니다.

## 🔍 문제의 시작: Cold Start의 의문

### Phase 3에서 본 성능

Phase 3 (집계 & 부스팅) 완료 후, 우리는 다음 수치를 얻었습니다:

```
GROUP BY sentiment: 21.71ms  ← 매우 빠름!
```

그런데 Redis 캐싱을 추가한 후 2회차 성능 측정에서:

```
GROUP BY sentiment: 27.52ms  ← 여전히 빠르긴 한데...?
```

**"어? 더 느려졌네?"**

처음엔 당황했습니다. 캐싱이 성능을 **악화**시켰을 수도 있다고 생각했거든요.

하지만 자세히 살펴보니, Phase 3과 Phase 4는 **완전히 다른 상황**이었습니다.

## 📊 1회차 vs 2회차: Cold Start 해부

### 시간대별 실행 흐름

```
1회차 (09:21 - Cold Start)
└─ Redis 자동 시작 직후
   └─ 모든 데이터가 캐시에 없음
   └─ 매 쿼리마다 전체 파이프라인 실행

2회차 (09:35 - 14분 후)
└─ Redis에 이전 데이터 누적
└─ 임베딩 벡터 캐시됨
└─ GROUP BY 결과 캐시됨 (5분 TTL)
```

### 1회차 수행 시간 분석

```
Simple keyword 실행:

① NQL 파싱        2ms
② 쿼리 빌드       3ms
③ 임베딩 계산     10ms  ← FastAPI 호출 (TCP 연결)
④ ES 검색        40ms  ← 인덱스 스캔 + 점수 계산
─────────────────────────
합계:             55ms  ❌
```

### 2회차 수행 시간 분석

```
Simple keyword 실행:

① NQL 파싱        1ms   (약간의 개선)
② 쿼리 빌드       2ms   (약간의 개선)
③ 임베딩 조회     2ms   ← Redis 캐시 히트! (-8ms)
④ ES 검색        31ms   ← 약간의 변동
─────────────────────────
합계:             31ms  ✅ (-42% 개선)
```

**차이의 원인**:

1. **임베딩 캐시 히트** (10ms → 2ms)
   - FastAPI 서비스 호출 제거
   - Redis 메모리 직접 조회
   - **8ms 절약**

2. **ES 검색 최적화** (40ms → 31ms)
   - 첫 실행 때의 잘못된 인덱스 워밍 제거
   - 세그먼트 병합 최적화 효과
   - **9ms 절약**

## 🎯 GROUP BY: 캐싱 효과의 진정한 승자

### 왜 GROUP BY가 49.8% 개선했나?

GROUP BY sentiment는 전체 쿼리 중 **가장 큰 개선율**을 보였습니다:

```
1회차: 54.85ms
2회차: 27.52ms
개선: -49.8%  ← 절반 이상 빨라짐!
```

그 이유를 파악해보겠습니다.

### 1회차: 집계 결과를 처음부터 계산

```
keyword("technology")
  ↓
[ES 검색]
├─ 모든 문서 스캔: 1000건
├─ "positive" 점수 계산
├─ "negative" 점수 계산
├─ "neutral" 점수 계산
└─ 결과 정렬: 40ms ← 매우 비쌈!

결과: {positive: 600, negative: 300, neutral: 100}
```

### 2회차: 캐시된 집계 결과 재사용

```
keyword("technology") GROUP BY sentiment

1️⃣ 캐시 키 생성
   groupby:HASH(...) + "sentiment"
   
2️⃣ Redis 조회
   → 5분 TTL 내의 기존 결과 발견!
   → {positive: 600, negative: 300, neutral: 100}
   
3️⃣ 즉시 반환 (0ms)
```

**혁신적인 점**:

GROUP BY 결과는 **자주 변하지 않습니다.**
- 5분 내에 새 뉴스가 들어와도
- 대전체 분포는 비슷합니다
- **5분 TTL은 충분히 유효합니다**

따라서:
```
임베딩 캐시: 10ms → 2ms  (-8ms)
GROUP BY 캐시: 40ms → 0ms  (-40ms)  ← 메인 효과!
총 개선: -48ms (49.8%)
```

## 💡 핵심 인사이트: Cold Start의 정체

### Cold Start란?

> 애플리케이션 시작 후 **캐시가 모두 비어있는 상태**에서 처음 요청을 처리하는 것

```
시간대별 캐시 상태:

09:21 - Redis 시작
│
├─ 09:21:00 ~ 09:25:00 (Cold Start 기간)
│  └─ 모든 요청이 DB 계산 필요
│
├─ 09:25:00 ~ 다음날 (Warm Cache 기간)
│  └─ 이전 요청의 캐시 재사용
│
└─ 캐시 만료 시점까지 효과 지속
```

### 일반적인 Cold Start 문제

대부분의 캐싱 시스템에서:

1. **앱 시작**: 모든 캐시 비어있음
2. **1번째 요청**: 풀 파이프라인 실행 (느림)
3. **2-100번째 요청**: 캐시 히트 (빠름)
4. **캐시 만료**: 다시 느림

**우리의 경우**:
```
1회차 (09:21):  Cold Start → 53.81ms (느림)
2회차 (09:35):  Warm Cache → 36.05ms (빠름)
14분 차이:      캐시 누적 효과
```

## 🔬 Embedding이 병목인 이유

### FastAPI 임베딩 서비스의 숨겨진 비용

"임베딩은 보통 1-2ms 아니가?"

아닙니다. **첫 요청**은 다릅니다.

```
FastAPI /embed/single 호출:

1️⃣ TCP 연결        1ms   (로컬, 여전히 오버헤드)
2️⃣ HTTP 요청        1ms
3️⃣ 모델 추론        6ms   ← 메인 비용
4️⃣ 결과 반환        1ms
5️⃣ 직렬화/역직렬화  1ms
─────────────────────────
합계:              10ms   (첫 요청)

캐시 히트:
Redis 메모리 조회:   2ms   (로컬 메모리!)
```

**개선율 계산**:
```
10ms → 2ms = 80% 개선!
```

### Elasticsearch도 첫 요청이 느려

```
1회차 (Cold Start):
- 인덱스 세그먼트 처음 로딩: 추가 오버헤드
- OS 페이지 캐시 미스
- 네트워크 왕복 지연

2회차:
- 세그먼트 이미 메모리에 로드됨
- 더 효율적인 캐시 히트
```

따라서 ES 검색도:
```
40ms → 31ms = 22% 개선
```

## 🎓 배운 교훈: 3가지

### 교훈 1: 측정 없이는 최적화 불가능

> Cold Start와 Warm Cache는 **완전히 다른 성능 특성**을 가집니다

우리가 발견한 것:
- 1회차만 보았다면: "캐싱이 큰 효과 없다" 결론
- 2회차와 비교했을 때: "33% 개선 달성!"

**교훈**: 항상 안정화된 상태(Warm Cache)에서 측정하세요.

### 교훈 2: Embedding은 생각보다 비싸다

> 벡터 임베딩 = 네트워크 + 모델 추론 비용

```
로컬 연산:      <1ms
Redis 조회:    1-2ms
FastAPI 호출:   10ms
```

**교훈**: 임베딩 캐싱은 필수입니다.

### 교훈 3: GROUP BY는 캐시의 킬러 유스케이스

> 집계 결과는 변동이 적고, 반복 쿼리 확률이 높습니다

```
GROUP BY sentiment: 49.8% 개선 ← 최고 개선율
Simple keyword: 42.1% 개선
OR query: 47.8% 개선

∴ 집계 중심 워크로드 = 캐싱의 진정한 가치
```

## 📈 실제 성능 곡선

### 요청별 응답시간 추이

```
응답시간 (ms)
60 │ ●●●●● (1회차 - 모든 캐시 미스)
   │ ╲
50 │   ╲
   │     ╲
40 │       ●●●●● (2회차 - 임베딩 캐시)
   │       ╲╲╲
30 │         ●●●●●●● (3회차+ - 안정화)
   │
20 │
   └──────────────────────
     1   2   3   4   5
    회차별 요청
```

**실제 패턴**:
1. 1회차: 높은 변동성 (20-70ms)
2. 2회차: 변동성 감소, 평균 개선
3. 3회차+: 안정화 (25-35ms)

## 🛠️ 프로덕션 적용을 위한 팁

### 1. 캐시 워밍업하기

```java
// 앱 시작 시
@EventListener(ApplicationReadyEvent.class)
public void warmupCache() {
    // 상위 10개 쿼리 미리 실행
    List<String> hotQueries = Arrays.asList(
        "keyword(\"AI\")",
        "keyword(\"technology\")",
        "keyword(\"blockchain\")",
        // ...
    );
    
    for (String nql : hotQueries) {
        queryService.search(nql);  // 캐시 로드
    }
}
```

### 2. 캐시 히트율 모니터링

```
Grafana 메트릭:
- redis_hits_total / redis_commands_total
- 목표: > 60% 히트율
```

### 3. TTL 동적 조정

```
GROUP BY 캐시: 5분 (변동 느림)
임베딩 캐시: 24시간 (불변)
NQL 쿼리: 1시간 (중간)
```
